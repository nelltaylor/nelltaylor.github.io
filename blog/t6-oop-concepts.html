Phase 0 Week 6 Technical Blog: Modules vs. Classes By: Nell Taylor | 11-14-14

Modules and classes in Ruby may seem very similar at first, as they can both be used to group methods and variables. While they both serve the purpose of operating in concert with other Ruby classes and objects to define and share their functionality, one of the easiest differences to remember to keep them straight is that a class can be instantiated while a module cannot. What this means is that since a class is a group of methods and variables that "model" a group of similar objects, you can instantiate a class &em; create an "instance" of it &em; which gives you a new object with all of the properties of that class that you can then add to and customize to suit that particular object. With a module, you cannot instantiate a new object from it, you can simply reference it in a class to extend the functionality of that module to the class. Where it gets even trickier is that classes can be part of a module's code. However, in order to include the functionality of a module within a class, the module file has to be linked to the class using either "load," "include", "require," or "extend." So what is a module useful for if you can't create new objects from it? A module is a series of methods and variables defined within a file separate from a single class so that the module's properties can be shared across multiple classes. A good example of why you might want to use a module would be something that changes the behavior of all classes it affects depending on what the method stored in it returns &em; an authentication, for instance. This common use of modules &em; to extend and mix in properties of a module as if it were actually a superclass of the class its affecting &em; is therefore called a mixin for a very good reason.
